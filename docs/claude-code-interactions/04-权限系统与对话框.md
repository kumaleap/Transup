# 04 权限系统与对话框交互

> 源码引用相对 `claude-code-analysis` 仓库根目录。

## 1. 总体架构：一次工具调用的权限链路

```
tool_use → hasPermissionsToUseTool()（纯判定，无 UI）
             ├─ allow → 直接执行
             ├─ deny  → 拒绝并回传 message 给模型
             └─ ask   → useCanUseTool → handleInteractivePermission
                          → push ToolUseConfirm 到 REPL 的 toolUseConfirmQueue
                          → <PermissionRequest> 按工具类型路由到专用对话框
                          → 用户选择 → onAllow(updatedInput, PermissionUpdate[], feedback?) / onReject(feedback?)
                          → persistPermissions()：应用到内存 context + 按 destination 写盘
```

关键文件：
- 判定核心：`src/utils/permissions/permissions.ts`
- 队列/回调装配：`src/hooks/useCanUseTool.tsx`、`src/hooks/toolPermission/handlers/interactiveHandler.ts`、`src/hooks/toolPermission/PermissionContext.ts`
- UI 路由：`src/components/permissions/PermissionRequest.tsx`
- 队列渲染与对话框优先级：`src/screens/REPL.tsx`

### 1.1 判定优先级（`hasPermissionsToUseToolInner`，permissions.ts ~L1158）

按顺序（先命中先返回）：

1. **1a. deny 规则**（整个工具被 deny）→ deny，`decisionReason: {type:'rule'}`
2. **1b. ask 规则**（整个工具被 ask）→ ask（例外：Bash 沙箱开启 autoAllowBashIfSandboxed 且命令可沙箱时穿透到 1c）
3. **1c. 工具自身 `tool.checkPermissions()`**（如 Bash 子命令级规则、文件路径规则），schema 解析失败则视为 passthrough
4. **1d. 工具级 deny** → deny
5. **1e. `requiresUserInteraction()` 的工具即使 bypass 也要 ask**（如 AskUserQuestion）
6. **1f. 内容级 ask 规则**（如 `Bash(npm publish:*)` 配置为 ask）→ **bypass 模式也不可跳过**
7. **1g. safetyCheck**（`.git/`、`.claude/`、`.vscode/`、shell 配置文件等敏感路径）→ **bypass 免疫，必须弹窗**
8. **2a. 模式放行**：`bypassPermissions` 或（`plan` 且 `isBypassPermissionsModeAvailable`）→ allow，`decisionReason: {type:'mode'}`
9. **2b. allow 规则**（整个工具被 allow）→ allow
10. **3. passthrough → ask**（默认弹窗）

外层 `hasPermissionsToUseTool` 再做模式变换：
- `dontAsk` 模式：ask → deny（附带固定拒绝文案）
- `auto` 模式（ant-only）：ask → 依次尝试 acceptEdits 快速通过 → 安全工具白名单 → YOLO 分类器，分类器拒绝则 deny 并弹 notification（`xxx denied by auto mode · /permissions`）

### 1.2 ToolUseConfirm 结构（PermissionRequest.tsx:103-127）

```ts
type ToolUseConfirm = {
  assistantMessage; tool; description; input; toolUseContext; toolUseID;
  permissionResult: PermissionDecision;       // 含 decisionReason / suggestions
  permissionPromptStartTimeMs: number;
  classifierCheckInProgress?; classifierAutoApproved?; classifierMatchedRule?;
  workerBadge?;                                // 队友/worker 来源徽标
  onUserInteraction();                         // 用户开始操作 → 取消异步自动批准
  onAbort();                                   // 中断整个 query
  onDismissCheckmark?();                       // 自动批准后的 ✓ 可用 Esc 立即消掉
  onAllow(updatedInput, permissionUpdates: PermissionUpdate[], feedback?, contentBlocks?);
  onReject(feedback?, contentBlocks?);
  recheckPermission(): Promise<void>;          // 模式切换后重查队列中挂起的请求
}
```

`interactiveHandler.ts` 细节：
- `onUserInteraction` 有 **200ms 宽限期**（防误触取消 classifier）
- `recheckPermission`：Shift+Tab 切模式后（`setToolPermissionContext` 触发），队列里的请求重新跑 `hasPermissionsToUseTool`，若变 allow 自动关闭弹窗放行
- 弹窗出现时通过 `useNotifyAfterTimeout(message, 'permission_prompt')` 发系统通知，文案按工具区分：ExitPlanMode→"Claude Code needs your approval for the plan"，EnterPlanMode→"…wants to enter plan mode"，默认→"Claude needs your permission to use {tool}"

## 3. 文件类权限对话框（FilePermissionDialog 家族）

目录：`src/components/permissions/FilePermissionDialog/`

### 3.1 选项生成 `permissionOptions.tsx` → `getFilePermissionOptions()`

固定三选项（value: `yes` / `yes-session`（或 `yes-claude-folder`）/ `no`）：

1. **Yes**（`accept-once`）— Tab 展开输入 "and tell Claude what to do next"
2. **会话级选项**（`accept-session`），文案按上下文变化：
   - 路径在 `.claude/`（项目或 `~/.claude/`）且非 read：`Yes, and allow Claude to edit its own settings for this session`
   - 工作目录内 + write：`Yes, allow all edits during this session (shift+tab)`（快捷键文本来自 `getShortcutDisplay('chat:cycleMode')`）
   - 工作目录内 + read：`Yes, during this session`
   - 工作目录外 + write：`Yes, allow all edits in {dirName}/ during this session (shift+tab)`
   - 工作目录外 + read：`Yes, allow reading from {dirName}/ during this session`
3. **No**（`reject`）— Tab 展开输入 "and tell Claude what to do differently"

### 3.2 决策处理 `usePermissionHandler.ts` → `PERMISSION_HANDLERS`
- `accept-once`：`onAllow(input, [], feedback)` — 不产生任何规则
- `accept-session`：
  - `.claude` scope：`addRules [{Edit, ".claude/**" 或 "~/.claude/**"}] allow destination:'session'`
  - 否则调 `generateSuggestions()`（`src/utils/permissions/filesystem.ts:1414`）：
    - write/create：`[{type:'setMode', mode:'acceptEdits', destination:'session'}]`（仅 default/plan 模式下才建议，避免降级 auto/bypass）+ 目录外时追加 `addDirectories(session)`
    - read + 目录外：`Read({dir}/**)` allow 规则（session）
- `reject`：`onReject(feedback)`

**全部是 session 级**——文件类对话框从不直接写 settings 文件。

### 3.3 useFilePermissionDialog.ts 的键盘交互
- 注册 `confirm:cycleMode`（Shift+Tab，Confirmation 上下文）：**直接等价于选中 accept-session 选项**
- Tab 切换 yes/no 的输入模式（`handleInputModeToggle`，带埋点）
- 焦点移走且无文字时收起输入

### 3.4 FilePermissionDialog.tsx 组件本体
- symlink 警告：write 且目标是符号链接时黄字 `This will modify {target} (outside working directory) via a symlink` 或 `Symlink target: {target}`
- IDE diff：连接 IDE 时 `useDiffInIDE` 在 IDE 内开 diff tab，终端切换为 `ShowInIDEPrompt`（`Do you want to make this edit to {file}?` + 相同三选项 + `Save file to continue…`；用户在 IDE 里改动 diff 会通过 `ideDiffSupport.applyChanges` 回写 updatedInput）
- 底部提示：`Esc to cancel`，聚焦 yes/no 且未展开输入时追加 ` · Tab to amend`
- Select 的 `onCancel`（Esc）→ 等同选 No（无反馈）

### 3.5 各文件工具的差异

| 组件 | title | question | diff 预览 |
|---|---|---|---|
| FileEditPermissionRequest | `Edit file`，subtitle=相对路径 | `Do you want to make this edit to {basename}?` | `FileEditToolDiff`（结构化 diff） |
| FileWritePermissionRequest | 文件存在→`Overwrite file`，否则 `Create file` | `Do you want to {overwrite\|create} …?` | `FileWriteToolDiff`（老内容 vs 新内容全量 diff） |
| NotebookEditPermissionRequest | 类似，languageName 取 cell_type | — | NotebookEditToolDiff |
| SedEditPermissionRequest | `Edit file`（sed -i 命令被解析成编辑） | 同 Edit | 应用 sed 替换后的 diff；`Pattern did not match any content` 兜底 |
| FilesystemPermissionRequest（Read/Glob/Grep） | `Read file`/`Edit file`（按 isReadOnly） | 默认 | content 为 `ToolName(参数渲染)` |

## 4. Bash 权限对话框

文件：`src/components/permissions/BashPermissionRequest/BashPermissionRequest.tsx` + `bashToolUseOptions.tsx`

- **标题**：`Bash command`；沙箱启用但该命令未沙箱时 → **`Bash command (unsandboxed)`**
- 内容：完整命令（verbose 强制 true）+ dim 的模型给出的 description + `PermissionRuleExplanation` + 可选**破坏性命令警告**（feature flag `tengu_destructive_command_warning`，黄字）
- **选项**（`bashToolUseOptions`）：
  1. `Yes`（Tab 可展开 accept 反馈输入）
  2. `Yes, and don't ask again for: [可编辑前缀输入框]`（value `yes-prefix-edited`，`type:'input'`、`showLabelWithValue`、separator `": "`）— 初值为前缀启发式（`getSimpleCommandPrefix` 两词 → `npm run:*`，否则一词，否则全命令）；复合命令用后端 tree-sitter 逐子命令建议（`decisionReason.type==='subcommandResults'`）；多条建议时退化为文本选项 `yes-apply-suggestions`（label 由 `generateShellSuggestionsLabel` 生成，剥离输出重定向文件名）
  3. （ant-only）`yes-classifier-reviewed`：自然语言描述规则输入框（"describe what to allow..."），持久到 **session** 的 prompt 规则
  4. `No`（Tab 展开 reject 反馈）
  - 仅当 `shouldShowAlwaysAllowOptions()`（无 `allowManagedPermissionRulesOnly` 管控）时显示 2/3
- **持久化**：`yes-prefix-edited` → `addRules Bash({prefix}) allow destination:'localSettings'`（写 `.claude/settings.local.json`）；`yes-apply-suggestions` → 直接应用 permissionResult.suggestions
- **键盘**：Esc=拒绝；Tab=amend；`ctrl+e` 切换 **Permission Explainer**（Haiku 生成"这条命令会做什么"的解释）；debug 模式下 `ctrl+d` 切换 `PermissionDecisionDebugInfo`
- **Bash classifier 异步自动批准**（ant-only）：弹窗副标题显示 shimmer 动画 `Attempting to auto-approve…`；期间用户任何交互（200ms 后）取消自动批准并显示 `Requires manual approval`；批准成功显示绿色 `✓ Auto-approved · matched "{rule}"`，选项全部禁用置灰，Esc 可立即消掉 checkmark；主 agent 有 2 秒 speculative classifier 竞速窗口（useCanUseTool.tsx），赢了就根本不弹窗

## 6. Permission Modes 与 Shift+Tab

### 6.1 模式定义（`src/utils/permissions/PermissionMode.ts`）

| mode | title | symbol | 颜色 | 外部暴露 |
|---|---|---|---|---|
| default | Default | （无） | text | default |
| plan | Plan Mode | ⏸（PAUSE_ICON） | planMode | plan |
| acceptEdits | Accept edits | ⏵⏵ | autoAccept | acceptEdits |
| bypassPermissions | Bypass Permissions | ⏵⏵ | error | bypassPermissions |
| dontAsk | Don't Ask | ⏵⏵ | error | dontAsk |
| auto（ant-only） | Auto mode | ⏵⏵ | warning | 映射为 default |

### 6.2 循环顺序（`getNextPermissionMode.ts`）
外部用户：`default → acceptEdits → plan → (bypassPermissions 若可用) → default`；bypass 之后回 default。`dontAsk` 不进循环。`isBypassPermissionsModeAvailable` 取决于启动时 `--dangerously-skip-permissions` / settings。

### 6.3 键位（`src/keybindings/defaultBindings.ts`）
- Chat 上下文：`shift+tab`（Windows 无 VT 模式时 `meta+m`）→ `chat:cycleMode`
- Confirmation 上下文：`shift+tab` → `confirm:cycleMode`（权限弹窗内直接选会话级选项）

### 6.4 切换处理（`PromptInput.tsx` `handleCycleMode` ~L1845）
`cyclePermissionMode()` 计算下一模式并做副作用（`transitionPermissionMode`：如进入 auto 时剥离危险 allow 规则 `Bash(*)` 等）→ `setAppState` + `setToolPermissionContext`（后者触发队列中弹窗 `recheckPermission`）→ 埋点 `tengu_mode_cycle`；进 plan 时写 `lastPlanModeUse` 到全局 config。首次切到 auto：先只改 UI 标签，400ms 防抖后弹 `AutoModeOptInDialog`，确认后才真正 transition。

### 6.5 视觉指示（`PromptInputFooterLeftSide.tsx` ModeIndicator）
非 default 模式在输入框下方渲染：`{symbol} {title小写} on (shift+tab to cycle)`，颜色用 `getModeColor(mode)`；提示在其他 footer 项 ≥2 时隐藏；远程模式不显示。

## 7. Plan Mode 完整流程

### 7.1 EnterPlanModePermissionRequest
- 边框色 `planMode`，标题 `Enter plan mode?`
- 正文：`Claude wants to enter plan mode to explore and design an implementation approach.` + 4 条 dim bullet（Explore/Identify/Design/Present）+ `No code changes will be made until you approve the plan.`
- 选项：`Yes, enter plan mode` / `No, start implementing now`；Yes → `onAllow({}, [{type:'setMode', mode:'plan', destination:'session'}])`

### 7.2 ExitPlanModePermissionRequest（~1150 行）
- 标题 `Ready to code?`（planMode 色）；空计划时简化为 `Exit plan mode?` + Yes/No
- 正文：`Here is Claude's plan:` + 虚线框内 Markdown 渲染计划 + 请求的 prompt 权限列表（ant-only）+ `…Would you like to proceed?`
- **选项构建 `buildPlanApprovalOptions`**（按可用性动态）：
  1. （若 settings `showClearContextOnPlanAccept`）`Yes, clear context (NN% used) and {use auto mode | bypass permissions | auto-accept edits}`（优先级 auto > bypass > acceptEdits）
  2. keep-context 提升模式：`Yes, and use auto mode` / `Yes, and bypass permissions` / `Yes, auto-accept edits`
  3. `Yes, manually approve edits`（→ default）
  4. （Ultraplan 可用时）`No, refine with Ultraplan on Claude Code on the web`
  5. `No, keep planning` — **type:'input'** 输入框，placeholder `Tell Claude what to change`，description `shift+tab to approve with this feedback`
- **键盘**：
  - `Ctrl+G`：在 $EDITOR 中编辑计划（V2 直接编辑 plan 文件），保存后显示 `✓Plan saved!` 5 秒；底部提示 `ctrl-g to edit in {editor} · {planFilePath}`
  - `Shift+Tab`：立即选 accept-edits 选项（即"带着 No 框里已输入的反馈批准"——反馈文本作为 acceptFeedback 传入 onAllow）
  - Esc：拒绝（留在 plan mode）
  - 支持向 No 输入框粘贴图片（作为 contentBlocks 随 onReject 回传）
- **状态流转**：
  - keep-context 批准 → `onAllow(updatedInput, buildPermissionUpdates(mode))`，即 `setMode(session)`（+ 可选 prompt 规则），并 `setNeedsPlanModeExitAttachment(true)`
  - clear-context 批准 → 不走 onAllow：`setAppState({initialMessage: {content: "Implement the following plan:…", clearContext:true, mode}})` 然后 `onReject()` 解锁 query loop，由 REPL 清上下文重新发起
  - `No` 且无反馈无图片 → 不动作（用户还在输入框上）
  - 自动以计划内容命名会话（Haiku）
- **fullscreen sticky footer**：计划太长时选项区通过 `setStickyFooter` 固定在底部

## 8. AskUserQuestion

目录：`src/components/permissions/AskUserQuestionPermissionRequest/`

- **结构**：多问题 → tab 式导航条（`QuestionNavigationBar`：`← [✓] Header1 [ ] Header2 … ✓ Submit →`，宽度自适应截断，当前 tab 优先）→ 每题 `QuestionView` → 全答完进 `SubmitQuestionsView`
- **状态机** `use-multiple-choice-state.ts`：`currentQuestionIndex / answers / questionStates{selectedValue,textInputValue} / isInTextInput`，reducer 驱动
- **选项渲染**：题目选项 → `Select`（单选，`layout="compact-vertical"`，description 在 label 下）或 `SelectMulti`（`question.multiSelect`）；末尾恒追加 **Other** 选项：`{type:'input', value:'__other__', label:'Other', placeholder:'Type something.'}`，聚焦 Other 时 `ctrl+g` 可开外部编辑器编辑长文本，支持粘贴图片（答案追加 `(Image attached)`）
- **multiSelect**：空格切换 `[✓]`，`submitButtonText`（最后一题 `Submit` 否则 `Next`）作为列表底部按钮，Enter 在按钮上才提交；答案为选中 label 以 `, ` join（`__other__` 替换为输入文本）
- **单题单选自动提交**：`questions.length===1 && !multiSelect` 时选中即 `submitAnswers`（跳过 review），且隐藏 Submit tab
- **preview 选项**：单选题选项带 `preview` 时切换为 `PreviewQuestionView` 左右分栏 carousel（左选项右预览，markdown 渲染，高度全局统一防抖动）
- **footer**（列表下方，↓ 从最后一项进入）：`{n+1}. Chat about this`（→ `onReject` 附带"用户想澄清问题"的结构化 feedback）；plan mode 下再加 `{n+2}. Skip interview and plan immediately`（→ onReject 附"停止提问直接完成计划"）
- **导航键**：`tabs:previous/next`（←/→、tab/shift+tab）切题；文本输入中禁用；Esc 取消（onReject 无反馈）
- **提交**：`onAllow(input+{answers, annotations{preview,notes}}, [], undefined, imageBlocks)`；帮助文案：`Enter to select · Tab/Arrow keys to navigate · Esc to cancel`
- **Review 页**：列出 `• 问题 → 答案`（绿色），未答完警告 `⚠ You have not answered all questions`，选项 `Submit answers` / `Cancel`

## 10. CustomSelect 组件（`src/components/CustomSelect/`）

- **选项类型** `OptionWithDescription<T>`：`text`（label/description/disabled/dimDescription）或 `input`（onChange/placeholder/initialValue/allowEmptySubmitToCancel/showLabelWithValue/labelValueSeparator/resetCursorOnUpdate）
- **键盘**（`use-select-input.ts`，导航走 keybinding 系统 Select 上下文）：
  - ↑/↓（及 ctrl+p/n）移动焦点；有 `onUpFromFirstItem/onDownFromLastItem` 时不回绕（AskUserQuestion 用它进入 footer）
  - Enter=`select:accept`（disabled 选项忽略）；Esc=`select:cancel`（自动注册 overlay 防止 Esc 被全局取消处理器截胡）
  - **数字键 1-9 直接选中对应项**（全角数字归一化）；目标是 input 选项时：已有预填文本→直接提交，allowEmptySubmitToCancel→提交空，否则仅聚焦进入输入
  - PageUp/PageDown 翻页；`visibleOptionCount` 默认 5，带上下箭头指示更多
  - **Tab** → `onInputModeToggle(focusedValue)`（权限对话框用于展开/收起反馈输入）
  - input 选项聚焦时：数字/字母直接进文本框，↑/↓ 仍导航，Enter 提交（空且不允许空→onCancel），ctrl+g 外部编辑器，↓ 可进入图片选择模式
- **渲染**：焦点行 `figures.pointer` 指针 + `suggestion` 色；已选 `success` 色；三种 layout（compact / expanded / compact-vertical）；`inlineDescriptions` 控制 description 内联
- **SelectMulti**（`SelectMulti.tsx` + `use-multi-select-state.ts`）：`[✓]` 复选框；有 submitButtonText 时 Enter=切换选中、按钮行 Enter=提交；无按钮时 Enter=提交、空格=切换

## 11. 决策持久化总表

`PermissionUpdate`（`src/utils/permissions/PermissionUpdateSchema.ts`）：`addRules / replaceRules / removeRules / setMode / addDirectories / removeDirectories`，destination ∈ `userSettings`（~/.claude/settings.json）/ `projectSettings`（.claude/settings.json）/ `localSettings`（.claude/settings.local.json，gitignored）/ `session`（仅内存）/ `cliArg`。

应用路径：`PermissionContext.persistPermissions()` → `persistPermissionUpdates()`（写盘，session/cliArg 跳过）+ `applyPermissionUpdates()`（更新内存 ToolPermissionContext，立即生效）。

| 用户选择 | 更新 | 目的地 |
|---|---|---|
| 文件类 "Yes, during this session" | setMode acceptEdits（±addDirectories/Read 规则） | session |
| .claude 文件夹选项 | Edit(.claude/**) allow | session |
| Bash "don't ask again for {prefix}" | Bash({prefix}) allow | **localSettings** |
| Bash classifier 描述规则 | Bash(prompt-rule) allow | session |
| WebFetch domain | WebFetch(domain:x) allow | **localSettings** |
| Fallback/MCP "don't ask again" | 整工具 allow | **localSettings** |
| Plan 批准 | setMode(acceptEdits/bypass/default) | session |
| Trust/Bypass/AutoMode/MCP 审批对话框 | 专用字段 | 项目 config / userSettings / localSettings |
| `allowManagedPermissionRulesOnly`（托管） | `shouldShowAlwaysAllowOptions()`=false → 隐藏所有 "don't ask again" 选项，session 级仍可用 | — |

## 12. Transup 实现要点（最小可行集）

1. 单一 `ToolUseConfirm` 队列 + `PermissionRequest` 路由器 + 每工具专用对话框；所有对话框输出统一收敛为 `onAllow(input, updates[], feedback?)` / `onReject(feedback?)`
2. 三段式选项模板：Yes（可 Tab 附言）/ Yes-scoped（会话或持久规则，文案含作用域）/ No（可 Tab 附言）；Esc=No；数字键直选；Shift+Tab 在弹窗内=选会话级选项
3. 判定顺序务必 deny > ask 规则 > 工具自检 > safetyCheck（bypass 免疫）> 模式 > allow 规则 > 默认 ask
4. 持久化区分 session（内存 context）与三层 settings 文件，"don't ask again" 默认写 local settings
5. 模式循环 default→acceptEdits→plan→(bypass)→default，footer 指示 `{symbol} {mode} on (shift+tab to cycle)`，切模式后对队列中弹窗做 recheck 自动放行
